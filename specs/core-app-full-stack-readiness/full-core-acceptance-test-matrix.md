# Full-Core Acceptance and Security Test Matrix

## Purpose

This is the authoritative acceptance and security checklist for generated **full core** secure AI-first SaaS apps in this migration. It consolidates the canonical core PRD, the full-core realization map, and the module slice specs into one generation-readiness test contract.

A plan, app description, or generated implementation is not full-core ready unless every `required` row below is covered by automated tests or by an explicitly recorded narrower-scope label such as `Module 1-only / not full core`.

## Coverage gates

| Gate | Required proof |
|---|---|
| Scope label | Full core artifacts state `full core`; any omission of User Admin, Agent Admin, invitation lifecycle, governed prompts/skills/manifests/tool boundaries, Audit/Trace, Governance/Policy, or workstream UI changes the label to a narrower non-full-core scope. |
| Backend authorization | Every protected endpoint, command, view query, workflow step, consumer side effect, timed action, stream, agent tool, and internal-agent operation resolves account, selected AuthContext, tenant/customer scope, role/capability, disabled status, and correlation id server-side. |
| Tenant/customer isolation | Cross-tenant/customer commands, queries, views, traces, streams, tools, workflows, timers, and UI actions deny without data leakage and emit caller-scoped audit/trace evidence. |
| Audit/trace evidence | Allowed, denied, no-op, idempotent, approval-required, failed, and sensitive-read paths emit durable AdminAuditEvent/AuditTraceEvent/WorkTrace or governed agent trace facts as specified. |
| Frontend boundary | Browser DTOs, fixtures, built JS/CSS/assets, and structured-surface payloads contain no WorkOS secrets, Resend secrets, model-provider secrets, raw JWTs, raw invitation tokens, token hashes, backend-only policy internals, or provider private ids. |
| Workstream UI | Functional rail visibility and action availability are backend-derived hints only; backend remains authoritative. Shell, composer, structured surfaces, states, accessibility, responsive behavior, and stale/reconnect paths are tested. |

## Minimum automated test families

Generated full-core implementations should organize automated coverage across these families:

1. **Domain/component unit tests** for Account/Profile/Settings/Membership/SupportAccess, Invitation, AgentDefinition, PromptDocument/Version, SkillDocument/Version, AgentSkillManifest, ToolPermissionBoundary, ModelConfigRef/Policy, AuditTraceEvent/WorkTrace, Evaluation/Rubric/Finding/Proposal/Decision/Outcome.
2. **Workflow tests** for invitation orchestration, approval/review, evaluation, activation, rollback, and any long-running behavior-change orchestration, including retries and idempotency.
3. **Consumer and timed-action tests** for email delivery/outbox, trace normalization/projections, version snapshot creation, invitation expiry/reminders, evaluation schedules, and stale callback no-ops.
4. **View tests** for tenant-scoped, paginated, redacted User/Admin/Invitation/Agent/Prompt/Skill/Manifest/Audit/Trace/Governance projections and query-shape constraints.
5. **HTTP/stream endpoint integration tests** using authenticated, unauthorized, forbidden, disabled, invalid-context, validation, conflict/idempotent, and cross-tenant cases.
6. **Governed agent runtime tests** with deterministic model providers for runtime resolution, prompt assembly, `readSkill`, tool-boundary enforcement, behavior-edit proposal flow, evaluator flow, traces, and denials before model invocation.
7. **Frontend tests/build** for typed API clients, state transitions, forms, validation mapping, workstream shell, structured surface rendering, realtime handling, accessibility, responsive behavior, static asset serving, and frontend secret boundary.
8. **Security review tests/checks** for backend authorization coverage, tenant/customer isolation, redaction, secret-never-store rejection, provider-secret boundaries, no frontend authorization reliance, approval/authority boundaries, and audit/trace completeness.

## Required full-core acceptance scenarios

1. Seeded Tenant Admin signs in through WorkOS/AuthKit, `/api/me` returns safe selected context, and the workstream shell opens.
2. Functional rail shows Access/Profile, User Admin, Agent Admin, Audit/Trace, and Governance/Policy only for authorized capabilities.
3. User Admin creates, resends, revokes, expires, and accepts invitations using local/dev/test captured outbox with audit through every transition.
4. Disabled account or membership cannot access protected UI, APIs, views, streams, tools, workflows, timers, consumers, or internal-agent operations.
5. Tenant/customer isolation denies access across all core resource families without leakage.
6. Agent Admin manages AgentDefinition, prompt, skill, manifest, tool-boundary, model ref, seed import, and deterministic test console lifecycles.
7. Runtime agents load skills only through authorized `readSkill(skillId)` and active manifests/tool boundaries.
8. Audit/Trace surfaces show correlated invitation, admin, prompt, skill, tool, model, agent-test, decision, and denial activity with redaction.
9. Governance/Policy turns an evaluator finding into a proposal, human review decision, activation or rejection, rollback path, and outcome observation.
10. Production-like frontend build and browser-facing APIs contain no backend/provider/auth/email/model/invitation secrets.
