# Catalog inventory: foundation workstream actions

Task: `TASK-WCTC-01-001`

Scope: docs/spec inventory only; no runtime catalog, backend, frontend, app-description, seed, or test code changed.

## Sources reviewed

- `WorkstreamService.runAction(...)`, `actionById(...)`, current `chatToolCatalog(...)`, plan proposal/confirmation/result DTOs, and `SurfaceAction` definitions.
- App-description workstream tool/surface catalogs under `app-description/domains/core-starter/workstreams/**`.
- Frontend workstream surface renderers under `frontend/src/workstream/surfaces/**`, especially `SurfaceRenderer.tsx` and `ChatToolPlanSurface.tsx`.
- Completed confirmed chat tool execution notes under `specs/workstream-chat-tool-execution/`.

## Classification key

- `chat-executable-now`: suitable for the next implementation pass as a confirmed `human_chat_tool_plan` entry because it has an existing backend-authorized `SurfaceAction`/`runAction` path, schema or intentionally read-only schema, selected `AuthContext` authorization, result surface, trace behavior, and can be covered by exact-snapshot/no-mutation/idempotency tests.
- `chat-proposal-only`: may create or update a draft/proposal/advisory task but must not commit final production authority or lifecycle state.
- `approval-gated`: may be proposed or routed only when a separate approval/decision/confirmation surface remains authoritative; chat confirmation alone is insufficient.
- `surface-only`: keep as structured surface action because target context, evidence review, field validation, or recovery UX should remain in the browser surface.
- `router-only`: deterministic no-mutation open/prefill route, not a chat tool execution step.
- `internal-only`: backend/service/support path not appropriate for direct chat catalog exposure.
- `blocked-pending-design`: missing prerequisite design/runtime policy/test coverage for safe chat exposure.
- `out-of-scope`: outside this five-workstream foundation catalog expansion.

## First expansion set recommendation

The first expansion set should stay narrow and additive. It should add mostly read/advisory/inert-draft actions where the existing surface/action path already carries selected-context authorization and result surfaces, plus a few low-risk idempotent commands. High-impact lifecycle, role, support-access, activation, rollback, export-delivery, and provider-backed autonomous outcomes remain approval-gated/surface-only/blocked.

Recommended first expansion set:

1. **My Account**: `action-update-my-profile`, `action-notification-mark-read`, `action-notification-dismiss`, `action-notification-archive`, `action-notification-snooze`, `action-notification-update-preferences`.
2. **User Admin**: `action-submit-user-admin-invitation`, `action-useradmin-resend-invitation`, `action-confirm-user-admin-invitation-revoke`, `action-submit-saas-owner-admin-invitation`, `action-customer-admin-invite`, `action-submit-customer-create`, `action-submit-customer-rename`, `action-submit-organization-rename`.
3. **Agent Admin**: no-side-effect/advisory task starts and proposal-submission surfaces only: `action-agent-detail-run-test`, `action-agent-prompt-governance-simulate`, `action-agent-skill-manifest-simulate`, `action-agent-tool-boundary-simulate`, `action-agent-model-refs-run-test`, `action-agent-prompt-governance-submit-review`, `action-agent-skill-manifest-submit-review`, `action-agent-tool-boundary-submit-review`, `action-agent-model-refs-submit-review`.
4. **Audit/Trace**: read/evidence actions plus note append: `action-audit-trace-search`, `action-audit-trace-detail`, `action-audit-trace-timeline`, `action-audit-trace-failure-evidence`, `action-audit-trace-investigation-guide`, `action-audit-trace-append-investigation-note`.
5. **Governance/Policy**: read/advisory/inert proposal actions: `action-governance-policy-list`, `action-governance-policy-read`, `action-governance-policy-draft-proposal`, `action-governance-policy-submit-proposal`, `action-governance-policy-simulate`, `action-governance-policy-start-impact-analysis`, `action-governance-policy-read-impact-analysis`.

Cross-cutting prerequisites for every selected entry:

- deterministic surface router still runs first and remains no-mutation;
- catalog entry names action id, browser tool id, governed tool id, capability id, input schema, idempotency policy, confirmation/approval behavior, result surface, and trace requirements;
- proposal generation fails closed when provider/runtime/tool-boundary configuration is unavailable and does not fabricate success;
- exact `planId + planSnapshotId + selectedContextId + requestedBy + confirmedBy + stepHashes` confirmation is required before any consequential command;
- selected `AuthContext`, tenant/customer scope, backend authorization, tool-boundary policy, idempotency, and trace emission are rechecked per step;
- tests include out-of-catalog denial, selected-context denial, no mutation before confirmation, idempotent replay, redaction/secret boundaries, and result/recovery surfaces.

## Cross-workstream frontend and test implications

Frontend support already has `ChatToolPlanSurface` rendering proposal, exact confirmation phrase, result, partial failure/recovery, and system-message states. Later implementation tasks should extend contract tests rather than add client-side planning heuristics. The browser must continue to:

- render no-mutation proposal copy and disable submit until the exact `CONFIRM <planSnapshotId>` phrase matches;
- submit the backend confirmation action only with selected context, plan id, snapshot id, step hashes, idempotency key, and correlation id;
- avoid editing plan inputs, hidden capabilities, tenant/customer scope, provider payloads, or policy gates;
- render completed/failed/skipped/recovery states without hiding committed steps;
- redact sensitive key names such as secret/token/JWT/provider payloads in step input summaries.

Expected test families for all future catalog slices:

- backend/API: catalog metadata completeness, prompt classification guardrails, router-first no mutation, exact confirmation, selected-context/capability denial, out-of-catalog denial, idempotency, partial failure/recovery, provider fail-closed, approval-gated results, trace events;
- frontend: plan proposal/result/system-message rendering, exact confirmation UX, no auto-submit, no `globalThis.confirm`, safe input redaction, accessibility/focus, partial failure/recovery display;
- app-description/spec: coverage map tying every accepted/blocked action to tool ids, capability ids, surface contracts, tests, and smoke evidence;
- security: tenant/customer isolation, hidden target no-enumeration, backend authorization, frontend secret boundary, provider/outbox/model fail-closed behavior.

## Residual risks for later tasks

- Several current actions use in-memory idempotent action result caching for direct actions; expanded multi-step chat execution must not claim durable runtime readiness until plan/step idempotency is durable enough for the stated scope.
- Read/search actions need careful prompt classification so they do not steal deterministic no-mutation surface routing or enumerate hidden targets.
- Role/status/support-access/lifecycle/activation/rollback/export actions are intentionally not in the first expansion set because they require richer approval, prerequisite, and recovery design.
- Provider/model-backed advisory tasks must not count fail-closed provider-unavailable states as successful planning or successful model analysis.
