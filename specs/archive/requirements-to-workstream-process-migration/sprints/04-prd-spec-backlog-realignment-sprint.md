# Sprint 04: PRD, Spec, and Backlog Planning Realignment

## Objective

Update direct planning paths so PRDs and requirements artifacts are decomposed into workstreams, attention models, dashboards, surfaces, capabilities, APIs, Akka components, autonomous tasks, events/notifications, and traces before implementation tasks are generated.

## Work areas

1. Update solution plan output sections to explicitly include workstream attention breakdowns, dashboard definitions, autonomous task candidates, notification mappings, and left rail/My Account aggregate implications.
2. Update PRD-to-specs flow so large PRDs split first by workstream or cross-workstream foundation, not CRUD module/component family.
3. Update sprint/backlog templates to produce vertical dashboard/surface/capability/autonomous-task increments.
4. Update revised PRD/change flows to preserve existing workstream/attention/dashboard/task IDs rather than regenerating broad component plans.
5. Update pending question generation to ask only when attention, authority, approval, autonomous task lifecycle, or surface/action behavior cannot be safely inferred.

## Acceptance criteria

- Direct planning paths produce implementation plans that match the target process contract.
- Backlogs preserve workstream, dashboard, surface, capability, API/exposure, Akka substrate, autonomous task, notification, attention, auth, audit, and test fields.
- Existing capability-first and secure foundation requirements remain mandatory.
