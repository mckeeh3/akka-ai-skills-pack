# Horizontal Implementation Map

This map is derived from functional agents, surfaces, and governed capabilities. It is not the primary app model.

Java backend realization keeps Akka's standard `ai.first.api.*`, `ai.first.application.*`, and `ai.first.domain.*` layers, then partitions ownership inside each layer: reusable platform/security/identity/agent-runtime code under `foundation`, built-in five-core-workstream code under `coreapp`, and user-owned domain extensions under `business.<area>`.

| Capability | Horizontal Akka/frontend candidates | Notes |
|---|---|---|
| `secure-tenant-user-foundation` | Event Sourced Entities for Invitation, Membership where full-core audit-grade history is required, AdminAuditEvent, and support-access lifecycle; Key Value Entities for Account, UserProfile, UserSettings, and current Role/Permission config where history is not required; Workflows for invitation delivery/acceptance and decision-card/approval handoff where risky User Admin actions require review; Views for `UserDirectoryView`, `MembershipView`, `InvitationView`, `AdminAuditView`, and selected `AccessReviewQueueView` behavior; Consumers for Resend email/outbox and audit enrichment; Timed Actions for invite expiry/reminders and support-access expiry; HTTP endpoints for `/api/me`, `/api/admin/users/dashboard`, `/api/admin/users`, `/api/admin/users/{accountId}`, invitation/membership/role/account/support/access-review/audit APIs; optional governed UserAdminAgent read/summarize tools and governed Resend `@FunctionTool` preview/send tools for authorized agents; React surfaces for Access/Profile, `user-admin-dashboard`, `user-admin-user-list`, and `user-admin-user-account`. | Preserve tenant/customer scope, backend auth, idempotency, and audit across every surface. The dashboard/list/detail User Admin flow must be view-backed: dashboard cards and list rows come from scoped views, and account detail opens through `findScopedUser`/view-derived ids rather than caller-supplied known user ids. First functional completion may make dashboard/list/detail read-only except that at least invitation resend/revoke or access-review decision-card creation must exercise a safe mutation/approval path with AdminAuditEvent output. |
| `ai-first-work-management` | Event Sourced Entities for Goal/Plan/Decision records; Workflows for long-running plans and approval waits; Agents for planning/summarization/evaluation; Views for mission control and progress; Timed Actions for reminders/rechecks; HTTP/SSE for workstream updates; React Governance/Policy and Governance Workbench surfaces. | Side-effecting agent tools default to proposal/approval. |
| `governance-decisions-audit` | Event Sourced Entities for Policy, ApprovalRequest, Decision, Audit/Trace records; Workflows for approval gates; Views for decision queue and trace search; Agents for proposal summaries and audit summaries; Consumers for trace enrichment; React decision-card, policy-governance, and audit-trace surfaces. | Human approval remains required for authority expansion and high-impact policy commits. |
| `frontend-shell-integration-patterns` | HTTP endpoints for browser APIs/SSE, static asset hosting for React/Vite build, typed frontend clients, shell components, surface renderers, route/deep-link handlers, frontend tests. | Routes open workstreams/surfaces; they do not define backend authority. |
| `managed-agent-foundation` | Event Sourced Entities for AgentDefinition and governed document/version lifecycle; Key Value Entities for active behavior pointer/cache where suitable; Workflows for proposal/review/activation/rollback; Agents for AgentBehaviorEditorAgent and evaluators; Views for catalogs/history/traces; HTTP endpoints for admin APIs and `readSkill`; React Agent Admin and Governance surfaces. | Prompt/skill content is behavior guidance only; ToolPermissionBoundary and backend AuthContext enforce authority. |
