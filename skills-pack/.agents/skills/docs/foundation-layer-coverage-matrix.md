# Foundation layer coverage matrix

Use this matrix to check whether the reusable horizontal SaaS Foundation App concepts are represented in the skills pack before planning or implementing domain-specific workstreams and agents.

The matrix maps foundation concerns to the installed skills-pack guidance that teaches them, the root app artifacts that currently realize them, and known coverage notes. It is not a runtime readiness claim by itself; runtime readiness remains governed by the root app `app-description/00-system/readiness-status.md` and `specs/full-core-saas-readiness/**` evidence.

## Coverage states

| State | Meaning |
| --- | --- |
| Covered | The concept has explicit skills-pack doctrine plus focused skill guidance and current root app/app-description anchors. |
| Partially covered | The concept is described, but implementation depth, examples, validation, or current-root alignment is intentionally incomplete or distributed. |
| Deferred in root app | The skills pack describes the foundation concept, but the current runnable root app explicitly defers full implementation for the current readiness scope. |
| Needs audit | Existing guidance may be stale, duplicated, or difficult to trace to current root implementation. |

## How to use this matrix for domain work

1. Classify the requested change by domain, workstream, surface, functional/internal agent, governed capability, and Akka substrate.
2. Check the relevant foundation rows above before adding app-specific behavior.
3. Load the smallest focused skill set for the touched rows instead of expanding into unrelated foundation work.
4. Record any intentionally deferred foundation effect in the target app-description/specs.
5. Validate through the real local Akka/API/UI path for the stated scope.

## Known alignment watchlist

- Root app readiness is current source of truth for what is implemented now; skills-pack docs can describe broader mandatory generated-SaaS expectations that are not fully implemented in the current root scope.
- Avoid references to old separate starter-template paths; the current model is root-app-first with `skills-pack/templates/ai-first-saas-core-app/**` as app-description guidance fragments.
- Keep `.agents/skills/**` as installed harness support. Do not treat it as writable app source or a duplicate baseline.
- When adding new horizontal foundation concepts, update this matrix, the relevant focused skill/docs, app-description anchors, and readiness evidence together.
