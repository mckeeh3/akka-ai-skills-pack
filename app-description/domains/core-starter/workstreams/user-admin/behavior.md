# Behavior: User Admin

## Current-state behavior

User Admin administers SaaS Owner Admin users, customer-facing Organizations, Organization Admin users for selected Organizations, Customers within Organizations, Customer Admin users for selected Customers, tenant/customer users, memberships, invitations, roles/capabilities, support access, access reviews, identity review/relink exceptions, and admin audit summaries within authorized app-owner/tenant/customer scope. The workstream starts from `surface-user-admin-dashboard`, accepts contextual composer requests, returns structured surfaces, and maps every consequential action to capability `user-and-access-administration` and governed backend tools.

## Deterministic responsibilities

Keep these responsibilities out of model-only behavior:

- Authorization: selected `AuthContext`, account status, disabled-user denial, membership, role/capability grants, tenant/customer scope, SaaS Owner scope, support-access visibility, and last-admin policy.
- First SaaS Owner Admin bootstrap: create or link only the initial owner admin from the current trusted bootstrap process using the provided deployment/local environment variable, reject public/browser/prompt bootstrap, enforce idempotency and already-bootstrapped denial, and emit immutable provenance/audit evidence.
- SaaS Owner Admin lifecycle: invite/list/read other SaaS Owner Admin users, validate `SAAS_OWNER_ADMIN` role assignment, enforce last-owner-admin and self-action protections, idempotency, delivery/outbox status, and audit.
- Organization lifecycle: list/read/create/rename/suspend/archive/reactivate customer-facing Organizations backed by Tenant boundaries without granting app-data/support/billing authority, with idempotency, safe no-op/conflict handling, cascade blocking/restoration semantics, terminal archive semantics, no hard-delete behavior, and audit. Archive is an explicit terminal destructive command (`saas_owner.organization.archive`), not a hidden variant of suspend; archived Organizations cannot be reactivated in the foundation app.
- Organization Admin lifecycle: bootstrap/invite/list/read/manage `TENANT_ADMIN` users for a selected Organization after that Organization exists, validate target Organization/Tenant, preserve actor SaaS Owner scope, enforce last-organization-admin protections, prevent SaaS Owner role assignment through Organization flows, and audit.
- Customer lifecycle: list/read/create/rename/update/suspend/archive/reactivate Customers within the selected Organization/Tenant, validate tenant ownership, prevent sibling-customer leakage, handle idempotency/no-op/conflict states, block/restore customer-scoped authority according to lifecycle state, and audit. Archive is an explicit terminal destructive command (`tenant.customer.archive`) that preserves audit/detail evidence but blocks reactivation and normal Customer/Admin/User operations.
- Customer Admin lifecycle: bootstrap/invite/list/read/manage `CUSTOMER_ADMIN` users for a selected Customer after that Customer exists, validate target Customer/Tenant, preserve actor Organization Admin scope, enforce last-customer-admin protections, prevent Organization Admin or SaaS Owner role assignment through Customer flows, and audit.
- Invitation lifecycle: email normalization, duplicate/open-invite handling, role validation, target scope validation, expiry, resend/revoke eligibility, standard invitee acceptance/onboarding, idempotency, outbox/Resend enqueue, provider delivery/bounce/failure status shaping, accepted account/membership linkage, immutable accepted evidence, and audit.
- Account and membership lifecycle: directory projection, zero-membership/no-selected-context recovery, account-level disable/reactivate across all memberships, membership-scoped disable/reactivate/suspend/remove semantics, provider identity deactivation/relink review, self-disable denial, last-admin protection, no-op/idempotent results, `/api/me` disabled-account versus no-active-membership responses, and audit.
- Role lifecycle: SMB role/capability matrix, preview delta, policy/approval requirement calculation, mutation idempotency, and last-admin preservation.
- Support access: only Tenant/Organization Admins may grant, extend, or revoke support access; grants may be tenant-wide or customer-scoped, use day-based maximum duration options, require purpose capture and approval from one Tenant/Organization Admin, enforce read-time expiry, refresh projection/attention where available, and emit audit/work traces.
- Access review: task lifecycle, evidence collection, provider/model readiness gates, progress/result shaping, and human result decision recording.
- Trace shaping: correlation ids, redaction, denial/event/result records, audit-work trace links, and browser-safe evidence summaries.

## Agent behavior

`user-admin-agent` may explain, summarize, draft, recommend, compare alternatives, ask clarifying questions, and prepare proposals/decision-card facts only within authorized capabilities. It cannot grant permissions through prompt text, bypass approval gates, mutate access autonomously, infer hidden evidence, or act outside its tool boundary. Model-backed turns use governed runtime configuration or fail closed.

## Access-review worker behavior

Internal/autonomous access-review work may collect scoped evidence, summarize risk, and produce recommendations with progress/result surfaces. Worker output cannot directly mutate memberships, roles, invitations, support access, identity links, or policy. Human acceptance/rejection records the review decision; follow-up changes route through deterministic User Admin capabilities and policy gates.

## Production runtime hardening behavior

- Invitation create and resend use the supported outbox/Resend delivery path when configured. Missing Resend or outbox configuration is fail-closed and returns provider/outbox blocked status; normal runtime must not report fake sent success. Invitee acceptance uses the standard onboarding path: signed invitation token validation, WorkOS/AuthKit authentication or signup for the invited email identity, backend target-scope and requested-role validation, account/profile link or creation, membership creation/activation for the invitation scope, accepted timestamp/status update, selected-context refresh eligibility, and audit/work trace emission. Revoked, expired, accepted, duplicate/open, stale, replayed, email-mismatched, disabled-account, and hidden/cross-scope invitations have explicit no-op, validation, conflict, or denial outcomes.
- Identity exception recovery is durable workflow/state behavior. Requests, review, approve, deny, recovery start, completion, failure, cancellation, stale conflict, and replay/no-op transitions are tracked with provider-boundary redaction and audit/work traces; raw WorkOS/JWT/provider payloads are never exposed to browser users or prompts.
- Access-review automation is model-backed in production. When active configuration exists it invokes a governed Akka Agent path with model policy, prompt/skill/reference governance where used, scoped evidence tools, ToolPermissionBoundary, and work traces. Missing model/provider/profile/boundary/governed-doc configuration is model-fail-closed and produces a blocker/system-message rather than deterministic normal success.
- Access-review results remain advisory. Human accept/reject records the review outcome only; membership, role, invitation, support-access, identity, or policy mutations still route through deterministic User Admin capability surfaces and approval gates.

## Edge cases

Repeated commands are idempotent where side-effecting. Stale data returns stale/reconnect or conflict state. Invitation acceptance by the same resolved invitee account for the same valid invitation returns the existing accepted membership/context rather than creating duplicates; acceptance by a different account, mismatched email identity, revoked/expired invitation, hidden target, disabled account, or stale token returns safe recovery without revealing hidden scope details. Creating an Organization does not implicitly create admins; Organization Admin bootstrap is a separate invite/management action that requires the created Organization/Tenant target. Creating a Customer does not implicitly create admins; Customer Admin bootstrap is a separate invite/management action that requires the created Customer target. Suspending an Organization/Tenant blocks tenant/customer work, new invitations, Customer creation, Customer Admin bootstrap, support-access grant/extend, managed-agent behavior activation, policy activation, and provider-backed advisory tasks in that Organization until reactivated; existing membership records remain audit-visible but cannot authorize work. Archiving an Organization is terminal for normal administration and additionally blocks reactivation, rename, Organization Admin bootstrap/maintenance, Customer creation/maintenance, descendant Customer selected contexts, and all tenant/customer work except authorized audit/detail reads; it also revokes pending invitations where safe, ends active support-access grants, cancels queued/running agent/access-review/policy tasks for the archived scope, marks source-linked notifications as source-resolved, and preserves authorized detail pages as read-only evidence. Suspending a Customer blocks only that customer-scoped authority and leaves sibling Customers and Organization Admin authority unaffected. Archiving a Customer is terminal for normal administration and blocks reactivation, rename, Customer Admin/User invite/manage, support-access grant/extend, customer-scoped notifications, and customer-scoped workstream openings while preserving authorized audit/detail visibility; it also revokes pending Customer-scope invitations, ends active Customer-scope support-access grants, cancels queued/running Customer-scope agent/access-review/policy tasks, marks Customer source-linked notifications as source-resolved, and preserves authorized detail pages as read-only evidence. Removing the final membership leaves the account in safe no-access recovery rather than deleting it; account disable blocks all memberships until account reactivation, while membership disable affects only the selected scope. Expired support access is denied at authorization/read time even before any dashboard projection refreshes. Hard deletion and provider erasure are unsupported unless a future current-intent update adds that lifecycle. SaaS Owner Admin invite/manage, Organization Admin invite/manage, and Customer Admin invite/manage flows must reject missing target scope, role escalation, last-admin loss, and hidden/cross-scope targets without leaking existence. Provider, model, outbox, or security misconfiguration returns actionable fail-closed feedback and traces without fixture/model-less success. Unsupported business-domain requests are routed to extension guidance rather than silently added. Direct API/deep-link attempts are authorized server-side and return safe `not_found_or_redacted`/forbidden system messages where appropriate.
