# Access: User Admin

## Authorized roles

- `app-admin`: app-owner level; manages App Admin users and Tenant Admin users for SMB tenant accounts only when represented by a backend-authorized SaaS Owner/App Admin `AuthContext`. Does not automatically manage tenant employees, Customer Admins, or Customer Users without an explicit backend-authorized selected context.
- `saas-owner-admin`: SaaS Owner selected context with backend `saas_owner.admin.*` may invite and manage other SaaS Owner Admins; with `saas_owner.organization.list/read` may list/read platform-safe Organizations; with `saas_owner.organization.create/rename/suspend/archive/reactivate` mapped to internal `saas_owner.tenant.manage` may create, rename, suspend, terminally archive, and reactivate eligible Organizations; with `saas_owner.organization_admin.*` may invite/bootstrap and manage `TENANT_ADMIN` users for a selected Organization/Tenant after that Organization exists. Organization is the browser/API term; backend authorization, audit, and state use Tenant boundaries. This role does not receive tenant/customer application-data access, support access, raw provider data, hidden counts, or billing-derived authority by implication.
- `tenant-admin`: Organization Admin / SMB tenant level; with `tenant.customer.*` may create and maintain Customers within the selected Organization/Tenant; with `tenant.customer_admin.*` may invite/bootstrap and manage `CUSTOMER_ADMIN` users for selected Customers after those Customers exist; also manages tenant employee access and is the only role allowed to grant, extend, or revoke tenant-wide or customer-scoped support access. Cannot manage sibling Organizations/Tenants, SaaS Owner/Admin users, Organization Admin peers beyond explicit tenant-admin-safe policy, or grant SaaS Owner authority.
- `customer-admin`: customer level; manages only selected Customer Users, invitations, customer roles, and customer access-review evidence permitted by customer policy. Cannot create sibling Customers, manage tenant employees, tenant-level roles/settings, other customers, Organization Admins, App Admin/Tenant Admin accounts, or grant/revoke support access.
- `auditor`: read scoped evidence and audit/access-review traces where permitted; no mutation.
- `saas-owner-support`: support-only level; may view or explain tenant/customer user data only through an active tenant-created support-access grant and audited selected context. This is separate from App Admin authority.

Role names are UI/intent labels only. Backend capability grants, selected `AuthContext`, membership state, approval policy, and resource ownership are authoritative.

## Scope rules

All reads, writes, surface actions, streams, agent turns, skill/reference loads, internal worker tasks, and governed-tool invocations use backend-owned selected `AuthContext`, tenant/customer ids, account status, membership status, role/capability grants, support-access state, resource ownership, and approval policy. SaaS Owner Organization administration calls require a SaaS Owner scoped selected context and map browser Organization ids to internal Tenant ids only inside backend-authorized service/API logic. SaaS Owner Admin user-management calls target SaaS Owner scoped memberships only. SaaS Owner Organization Admin user-management calls target `TENANT_ADMIN` memberships/invitations inside one selected Organization/Tenant while preserving the actor's SaaS Owner scope and prohibiting tenant app-data access by implication. Customer management calls require a Tenant/Organization Admin selected context and target Customers inside that selected tenant. Organization Admin Customer Admin user-management calls target `CUSTOMER_ADMIN` memberships/invitations inside one selected Customer while preserving the actor's Tenant/Organization Admin scope and prohibiting sibling-customer, Organization Admin, or SaaS Owner authority by implication.

Frontend rail visibility, disabled buttons, hidden actions, prompt text, expertise text, and client filters never grant authority. Direct API/deep-link requests still receive backend authorization and safe denials.

## Required denials

Server-side denials include disabled actor, inactive membership, missing selected context, missing capability, SaaS Owner Admin without SaaS Owner context, SaaS Owner Admin missing `saas_owner.admin.*`, `saas_owner.organization.*`, `saas_owner.organization_admin.*`, or mapped internal `saas_owner.tenant.read/manage`, App Admin attempting tenant/customer user management without explicit selected-context authority, Tenant Admin attempting SaaS Owner Admin, App Admin, sibling Organization, or cross-tenant administration, Tenant Admin missing `tenant.customer.*` or `tenant.customer_admin.*`, Tenant Admin attempting sibling-customer administration, Customer Admin tenant-level, Organization Admin, customer creation, or other-customer action, SaaS Owner without support grant, Organization Admin attempting app-data/support-access/billing-derived access beyond explicit selected context, SaaS Owner assigning `SAAS_OWNER_ADMIN` through an Organization/Tenant admin flow, Organization Admin assigning `TENANT_ADMIN` or `SAAS_OWNER_ADMIN` through a Customer Admin flow, Organization Admin invite before Organization/Tenant exists, Customer Admin invite before Customer exists, cross-tenant/customer target, hidden/not-found target, role escalation, last SaaS Owner Admin loss, last Organization Admin loss, last Customer Admin loss, last tenant/customer admin loss, self-disable/self-admin-role-removal, support-access policy violation, identity relink policy denial, unsupported bulk side effect, unredacted audit export, provider/model/outbox unavailable, missing tool-boundary grant, and denied skill/reference load.

Denials produce safe `system-message` feedback where user-facing, emit audit/work traces, and must not reveal protected identities, hidden counts, cross-scope existence, raw provider ids, raw invitation tokens, secrets, or private evidence.

## Approval and confirmation

Organization create, rename, suspend, archive, and reactivate require explicit human confirmation, idempotency key, reason where consequential, correlation id, and backend audit/work trace emission; Organization archive requires terminal consequence confirmation and cannot be reactivated by the foundation app. Customer create, rename/update, suspend, archive, and reactivate require explicit human confirmation, idempotency key, reason where consequential, correlation id, selected Organization/Tenant proof, and backend audit/work trace emission; Customer archive requires terminal consequence confirmation and cannot be reactivated by the foundation app. SaaS Owner Admin invitation/management, Organization Admin invitation/bootstrap/management, and Customer Admin invitation/bootstrap/management require explicit human confirmation, idempotency key, target scope proof, role validation, delivery/outbox readiness for invitations, and backend audit/work trace emission. Invitation send/resend/revoke, membership status changes, role changes, Tenant Admin support-access grants/revocations/extensions, identity relink, and access-review resolution require explicit human confirmation and/or decision-card approval according to policy. Support access may target tenant-wide or customer-scoped access, uses day-based duration options, and requires approval from one Tenant/Organization Admin. Last SaaS Owner Admin loss, last Organization Admin loss, last Customer Admin loss, last tenant/customer admin loss, role escalation, support-access expansion, identity relink/reset, bulk operations, and low-confidence recommendations are risky by default.
