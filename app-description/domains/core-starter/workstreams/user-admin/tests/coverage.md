# Tests: User Admin

## Acceptance

- Given an authorized caller with selected `AuthContext`, when they open User Admin, then `surface-user-admin-dashboard` renders scoped data, safe attention queues, trace refs, and only authorized actions, including SaaS Owner Admin and Organization/Organization Admin branch actions only for authorized SaaS Owner contexts.
- Given dashboard queue filters or attention cards, when a user opens a dashboard object, then the backend-authored target is `surface-user-admin-users`, `surface-user-admin-user-detail`, `surface-user-admin-invitation-detail`, `surface-user-admin-role-change-preview`, `surface-user-admin-support-access-grant`, `surface-user-admin-support-access-revoke-confirmation`, `surface-user-admin-access-review-task`, `surface-user-admin-identity-exception-review`, Audit/Trace, or `surface-user-admin-system-message` according to the object state, with no cross-scope leakage.
- Given an authorized dashboard or users-directory create/invite action, when activated, then it opens `surface-user-admin-invitation-create` with selected-scope role options and does not create a user until the create form is submitted with idempotency/correlation data.
- Given a scoped directory row/card, when activated by click or keyboard, then backend-authored row state routes to the correct target: active/disabled/suspended members open user detail, invitation-only/failed/expired invite rows open invitation detail, role proposals open role-change preview, access-review rows open access-review task, identity exception rows open identity-exception review, and hidden/stale/forbidden rows open system-message. The list does not perform consequential mutation inline.
- Given an allowed invitation action, when opened/submitted with valid input and idempotency/correlation data, then the create, resend-confirmation, revoke-confirmation, or detail surface returns updated structured surfaces, outbox/Resend boundary state, no-op/denial states, and audit/work traces.
- Given an invitee follows a valid delivered invitation link, when WorkOS/AuthKit authentication or signup resolves the invited email identity and the backend validates the signed token, target scope, requested role, invitation status, expiry, and account eligibility, then acceptance creates or links the account/membership, marks the invitation accepted, returns selected-context refresh guidance, and emits audit/work traces without exposing raw tokens.
- Given an allowed membership/status/role/support-access action, when opened/submitted, then dedicated membership lifecycle, role preview, support grant, or support revoke surfaces enforce backend policy and produce success, no-op, approval-required decision card, validation, denial, stale/conflict, or failure with trace refs.
- Given a User Admin agent guidance request, when provider/model/security configuration is valid, then the governed runtime uses scoped evidence/tool boundaries and returns structured guidance; when config is missing, it fails closed with a blocked system message and trace.
- Given no active SaaS Owner Admin exists, when the trusted bootstrap configuration or migration/seed path runs, then it creates or links only the first SaaS Owner Admin, records provenance/audit evidence, is idempotent for the same configured principal, and rejects public/browser/prompt/fixture bootstrap attempts or additional owner-admin creation after bootstrap.
- Given a SaaS Owner Admin with `saas_owner.admin.*`, when they list, invite, resend/revoke invitations for, or manage another SaaS Owner Admin, then app-owner scoped surfaces/API responses are browser-safe, idempotent, audited, and enforce last-owner-admin and self-action protections.
- Given a SaaS Owner Admin creates an Organization, when they bootstrap the first Organization Admin, then the invite targets that created Organization/Tenant, assigns only Organization Admin-safe roles such as `TENANT_ADMIN`, queues through the invitation/outbox path, and returns Organization Admin detail/invitation status without granting tenant app-data access.
- Given a SaaS Owner Admin manages Organization Admins for an existing Organization, when they list, invite, resend/revoke invitations, change roles, suspend/reactivate, or remove membership, then backend authorization validates the target Organization/Tenant, enforces last-organization-admin protection, denies SaaS Owner role assignment through this flow, and emits audit/work traces.
- Given an Organization Admin creates a Customer, when they bootstrap the first Customer Admin, then the invite targets that created Customer, assigns only Customer Admin-safe roles such as `CUSTOMER_ADMIN`, queues through the invitation/outbox path, and returns Customer Admin detail/invitation status without granting sibling-customer, Organization Admin, or SaaS Owner authority.
- Given an Organization Admin manages Customer Admins for an existing Customer, when they list, invite, resend/revoke invitations, change roles, suspend/reactivate, or remove membership, then backend authorization validates the target Customer/Tenant, enforces last-customer-admin protection, denies Organization Admin/SaaS Owner role assignment through this flow, and emits audit/work traces.
- Given an access-review task, when started/read/cancelled/accepted/rejected, then task progress/result/review surfaces render and no worker output directly mutates access.

## Security and negative

- Disabled users, inactive memberships, zero-membership accounts, missing selected context, missing capability, Customer Admin tenant-level actions, Tenant Admin sibling-Organization or SaaS Owner Admin actions, Tenant Admin sibling-customer actions, SaaS Owner without support grant, SaaS Owner Organization Admin actions without SaaS Owner context, Organization Admin Customer actions without Tenant context, cross-tenant/customer requests, Organization Admin invite before Organization exists, Customer Admin invite before Customer exists, role escalation, SaaS Owner role assignment through Organization Admin flows, Organization Admin/SaaS Owner role assignment through Customer Admin flows, invitation acceptance with malformed/expired/revoked/stale/already-accepted-by-another-account tokens, invitee email mismatch, disabled account, hidden target, suspended/archived Organization or Customer, unsupported requested role, last-owner-admin loss, last-organization-admin loss, last-customer-admin loss, last-admin loss, self-disable, support-access policy violations, identity relink denials, unsupported bulk actions, public self-registration, hard deletion attempts, and hidden/not-found targets are denied without protected-data leakage.
- Agent/tool calls cannot exceed governed tool boundary, manifest assignment, skill/reference status, redaction/token limits, model policy, or approval policy.
- Browser payloads never expose raw JWT/session data, WorkOS/provider internals, invitation tokens/token hashes, Resend/provider secrets, full email bodies, hidden authority state, raw model/provider config, fixture data, or cross-scope counts.

## Surface rendering

- Dashboard, SaaS Owner Admin directory/invite, Organization directory/detail/create/rename/lifecycle, Organization Admin directory/invite/detail/manage, Customer directory/detail/create/rename/lifecycle, Customer Admin directory/invite/detail/manage, users directory, user detail, invitation create/detail/resend-confirmation/revoke-confirmation, membership lifecycle confirmation, role-change preview, support-access grant/revoke, access-review task, identity-exception review, decision-card, markdown-response, audit-link, and system-message variants render loading, empty, ready, submitting, validation-error, forbidden, not_found_or_redacted, conflict, stale/reconnect, partial-data, provider/model/outbox fail-closed, no-op, approval-required, and failure states.
- SaaS Owner Admin, Tenant Admin, Customer Admin, SaaS Owner support, Auditor, disabled-user, forbidden, redacted, empty, stale, and error variants show correct scope labels, queues, rows, redactions, actions, and denials. Tenant/Customer contexts omit SaaS Owner Admin and Organization branch actions unless a direct attempt returns safe denial.
- Responsive table-to-card fallback preserves authority, role, invitation, support-access, access-review, identity-exception, risk, trace, create/invite, and decision-card affordances; each card remains a discovery/selection control that opens the backend-authored target inspection or task surface.

## Production runtime hardening scenarios

- Invitation delivery: configured captured-provider tests prove create/resend queues through the outbox/Resend path, records delivery attempts/status/audit/work traces, ingests safe delivery/bounce/failure callbacks, and refreshes invitation detail; missing Resend config returns fail-closed provider/outbox system-message without fake success; failure/retry/replay/revoke/expired/accepted/superseded paths are idempotent and safe, and accepted invitation evidence is immutable.
- Invitation acceptance/onboarding: valid signed-token + WorkOS/AuthKit invitee authentication creates or links the account/membership for SaaS Owner Admin, Organization Admin, Customer Admin, tenant employee, and Customer User invitation scopes as applicable; repeated completion by the same resolved account is idempotent; `/api/me`/selected-context refresh shows the new authorized context without granting unrelated scope.
- Invitation security: cross-scope targets, hidden invitations, duplicate/open invites, disabled actors, missing capability, malformed/expired/revoked/stale/already-accepted tokens, invitee email mismatch, hidden target, suspended customer, and raw-token/full-email/provider-secret requests are denied or redacted without exposing tokens, provider ids, secrets, hidden counts, or full email bodies to browser payloads.
- Identity exception recovery: request, review, approve, deny, recovery start, completion, cancellation, provider failure, stale conflict, and replay/no-op are covered as durable lifecycle transitions with provider-boundary redaction and audit/work trace evidence.
- Identity security: hidden/cross-scope targets, denied identity relink policy, raw WorkOS/JWT/provider payload requests, and unauthorized recovery attempts return safe `system-message` or workflow-status results without existence leakage.
- Model-backed access review: configured test-model/provider path invokes the governed Akka Agent runtime, enforces model policy and ToolPermissionBoundary, emits prompt/model/tool/data/policy/result traces, and returns advisory progress/result surfaces.
- Access-review fail-closed/security: missing model/provider/profile/boundary/governed-doc config, denied evidence access, denied skill/reference load, and cross-scope evidence attempts produce blocker/system-message states; agent results cannot directly mutate access and require explicit human accept/reject review.
- Organization/Customer lifecycle cascade: suspending an Organization blocks tenant/customer work, invitations, Customer creation, Customer Admin bootstrap, support-access grant/extend, managed-agent/policy activation, and provider-backed advisory tasks until reactivation; suspending a Customer blocks only that customer-scoped authority and preserves sibling-customer/Organization Admin boundaries; archived and hard-delete attempts return safe denied/unavailable states.
- Account deprovisioning: removing the last membership leaves the account in no-access recovery for `/api/me`/My Account, provider identity deactivation routes to disabled/identity-exception recovery, and permanent deletion/anonymization/provider-erasure requests are denied or deferred as unsupported without inventing purge behavior.
- Frontend/secret boundary: User Admin surfaces render invitation delivery, identity exception, access-review result, and blocked system-message states without exposing Resend keys, raw invitation tokens, WorkOS/JWT payloads, raw prompts, model provider internals, hidden evidence, or unredacted trace detail.

## Idempotency and observability

- Repeated side-effecting actions do not duplicate effects.
- Denials, approval-required outcomes, no-ops, provider/model/outbox blocked states, prompt assembly, skill/reference loads, tool calls, data access, decision cards, and trace emissions are verifiable through local Akka/API/UI tests or readiness evidence.
