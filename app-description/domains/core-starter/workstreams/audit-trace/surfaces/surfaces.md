# Surfaces: Audit/Trace

Audit/Trace is the cross-workstream investigation workstream for scoped audit events, work traces, authorization denials, provider/model/tool failures, policy decisions, governed agent evidence, support-access evidence, export decisions, and correlation timelines. It owns `audit-trace-agent`; other workstreams may link to Audit/Trace evidence, but trace links are only references until reauthorized by Audit/Trace capabilities.

## Surface bindings

| Surface id | Type | Surface contract | Purpose |

## `surface-audit-trace-dashboard` contract details

- Surface role: workstream dashboard and action router for Audit/Trace investigations. The dashboard does not render passive metrics; every card, counter, row, badge, shortcut, and queue either opens a typed Audit/Trace surface through a governed backend action or is omitted when the actor is not authorized.
- Owner: Audit/Trace workstream, functional agent `audit-trace-agent`; reusable placement from other workstreams is by trace reference only and must reauthorize through Audit/Trace capabilities before dashboard data or drilldowns are returned.
- Actors and scope: tenant admins, tenant auditors/compliance reviewers, authorized support operators, and customer-scoped support actors with selected `AuthContext`, tenant id, optional customer scope, role/capability summary, and active support-access basis where required. Customer-scoped actors see only customer-authorized redacted counts and links. Unauthorized, disabled-user, expired-support-access, or cross-tenant/customer contexts receive `forbidden` or `not_found_or_redacted` outcomes without hidden ids, hidden counts, or policy internals.
- User goal: decide what investigation work needs attention now, open the least-privilege evidence surface, and request safe follow-up work without expanding authority or exposing raw evidence.
- Primary decision/action: choose an attention item or action shortcut. Success means the actor lands on an authorized search, detail, timeline, failure-evidence, investigation-guide, export-request, summary-progress, or safe system-message result with preserved tenant/customer scope and trace/correlation context.

### Dashboard payload schema

Top-level browser-safe fields for `audit.trace.dashboard.v1`:

- `surfaceContract`, `surfaceId`, `generatedAt`, `selectedScope`, `authContextSummary`, `capabilityIds`, `correlationId`, `traceRefs`, `redaction`, `readiness`, `sections`, `cards`, `attentionItems`, `actions`, `emptyState`, and `recovery`.
- `selectedScope`: tenant display label, optional customer display label, scope kind, and support-access status; never raw session data, hidden tenant/customer ids, or provider identifiers.
- `authContextSummary`: actor type, role labels, support/auditor/admin basis, and capability labels; implementation role ids and policy clause ids are diagnostic-only and never shown in the default view.
- `redaction`: summary counts and omitted field keys by category, plus safe explanation text; no raw evidence bodies, prompts, model/provider credentials, invitation tokens, or cross-scope details.
- `readiness`: `ready`, `partial`, `blocked-provider-or-runtime`, or `forbidden`, with safe recovery text and trace refs.
- `sections[]` order is fixed: first `needs-attention`, then `things-i-can-do`, then optional `recently-recorded`/`readiness-notices` if they contain clickable result targets. Non-actionable FYI metrics are excluded.
- `cards[]`: stable `cardId`, label, safe count/status/severity, short explanation, `actionId`, governed capability, result surface id, keyboard label, `emptyBehavior`, and optional redaction badge. Valid zero-count cards open an empty authorized queue or explanation surface.
- `attentionItems[]`: stable `itemId`, event category, severity, workstream/source label, safe actor label, age/status, redacted summary, `actionId`, result surface id, trace ref, and recovery/next-step text.
- `actions[]`: stable action id, browser label, capability id, approval requirement if any, idempotency/correlation expectation, result surface/system message, and hidden/denied reason category for audit logs. Hidden actions are normally omitted from the default dashboard.

Default visible content is limited to investigation summaries, counts, severity/status, user-actionable next steps, scope labels, and redaction explanations. On-demand role-gated drilldowns may show diagnostic trace ids, policy refs, provider/model/tool failure categories, and support-access basis. Internal-only metadata includes raw auth tokens, raw provider/model/tool payloads, raw prompts, hidden policy implementation ids, backend component names, full event bodies, and correlation/idempotency internals beyond the safe correlation label.

- States: loading, empty, ready, submitting, validation-error, forbidden, `not_found_or_redacted`, stale/reconnect, partial-data, provider-fail-closed, model-fail-closed, tool-boundary-denied, approval-required, no-op/idempotent replay, recorded, blocked-provider-or-runtime, and failure. Every state preserves selected scope, redaction summary, recovery text, and trace/correlation refs when allowed.
- Audit/work traces: dashboard reads, denials, partial redaction, search/detail/timeline/failure/export/summary/note actions, idempotent replays, approval-required decisions, provider/model/tool-boundary failures, and no-op outcomes emit audit/work trace refs. Dashboard trace links are summaries until reopened and reauthorized through Audit/Trace capabilities.
- Accessibility/responsive semantics: implement with the selected web UI style guide `.agents/skills/docs/web-ui-style-guide.md`, component catalog anatomy `.agents/skills/docs/web-ui-component-catalog.md`, and named theme `core-saas-foundation`. Cards and rows are keyboard-operable controls with visible focus, ARIA labels based on safe labels, responsive single-column grouping on narrow viewports, and no color-only severity communication.
- Required app-description tests: acceptance coverage for authorized dashboard load and each visible action result; regression coverage for zero-count actionable cards and stale/reconnect recovery; security coverage for missing bearer, disabled user, role denial, support-access expiry, customer-scope restriction, cross-tenant/cross-customer redaction, hidden count non-enumeration, and frontend secret boundaries; negative coverage for invalid filters, hidden trace ids, unauthorized export, and model/provider/tool-boundary fail-closed states; idempotency coverage for note append/export/summary-start replays; observability coverage that allow/deny/no-op/failure paths record audit/work traces and preserve correlation refs; UX usefulness coverage that the default dashboard answers “what needs my attention” before “what can I do."
- Surface-description sufficiency review: this dashboard definition is sufficiently unambiguous for implementation and review without inventing payload fields, actions, states, auth/tenant behavior, trace links, tests, or visual/component semantics. No additional description pass is required for the `fully-specified` objective.

## `surface-audit-trace-search` contract details

- Surface role: list-search surface for scoped Audit/Trace evidence discovery. It returns redacted, row-level investigation results only; opening evidence, timelines, failure detail, export requests, or guidance must happen through governed backend actions and result surfaces.
- Owner: Audit/Trace workstream, functional agent `audit-trace-agent`. Reusable placement from My Account, User Admin, Agent Admin, and Governance/Policy is by trace/correlation/filter reference only and must reauthorize through Audit/Trace capabilities before any row or count is returned.
- Actors and scope: tenant admins, tenant auditors/compliance reviewers, authorized support operators, and customer-scoped support actors with selected `AuthContext`, tenant id, optional customer scope, role/capability summary, and active support-access basis where required. Unauthorized, disabled-user, expired-support-access, or cross-tenant/customer contexts receive `forbidden` or `not_found_or_redacted` outcomes without hidden trace ids, hidden row counts, raw policy internals, or source evidence bodies.
- User goal: find the least-privilege redacted trace/event evidence relevant to an investigation, then open an authorized detail, timeline, failure-evidence, investigation-guide, or redacted export request without expanding authority.
- Primary decision/action: submit or refine scoped filters and select an authorized result row or result-level action. Success means the actor receives `audit.trace.search.v1` rows and summary counts that are safe for the selected scope, or a typed empty/partial/denied state with recovery text and trace/correlation refs.
- Data source: backend-owned Audit/Trace query path over audit events, work traces, agent traces, denials, provider/model/tool failures, policy decisions, support-access evidence, export decisions, and correlation metadata already retained for the selected tenant/customer scope. The browser must not search local caches, hidden fixture data, raw provider payloads, or frontend-derived authority.

### Search payload schema

Top-level browser-safe fields for `audit.trace.search.v1`:

- `surfaceContract`, `surfaceId`, `generatedAt`, `selectedScope`, `authContextSummary`, `capabilityIds`, `correlationId`, `traceRefs`, `redaction`, `readiness`, `query`, `filters`, `resultSummary`, `rows`, `pageInfo`, `partial`, `actions`, `emptyState`, `validationErrors`, and `recovery`.
- `query`: browser-entered text after backend validation, normalized display value, applied filter chips, and safe search help text. It must never echo raw prompt text, raw request/response bodies, tokens, provider identifiers, hidden policy ids, or unredacted event bodies.
- `filters`: time window, evidence category, workstream/source surface label, actor label/type, severity, status/decision, correlation label, customer scope label when authorized, and redaction/category toggles. Hidden tenant/customer ids, raw principal ids, internal component names, and provider/model/tool payload identifiers remain internal-only.
- `resultSummary`: total visible result count or capped/approximate count when exact counts would reveal hidden evidence, applied scope label, omitted category counts by safe category, partial-data reason, and user-actionable next step text.
- `rows[]`: stable backend result key, timestamp/age, safe trace ref label, correlation label, event kind/category, source workstream/surface label, safe actor label, severity, status/decision, redacted summary, redaction badges, evidence category badges, available row action ids, and recovery text for partially hidden rows. Rows are clickable/keyboard-operable only when an authorized result surface is available.
- `pageInfo`: page size, cursor or page token label, has-more flag, sort label, and stale/reconnect hints; raw database cursors or retained event ids are not browser-visible.
- `partial` and `redaction`: omitted categories, omitted field keys, hidden-count policy, customer/support-scope restrictions, safe explanation, and trace refs proving that redaction was applied.
- `actions[]`: stable action id, browser label, capability id, target/result surface, approval requirement if any, correlation expectation, and hidden/denied reason category for audit logs. Hidden actions are omitted from default rows rather than displayed as disabled controls unless the denial itself is the safe outcome.

Default visible content is limited to scoped investigation summaries, labels, severity/status, timestamps, redaction explanations, and next-step actions. Role-gated drilldowns may expose diagnostic trace ids, correlation labels, support-access basis, and policy refs through result surfaces. Internal-only metadata includes raw event bodies, prompts, provider/model/tool payloads, hidden tenant/customer identities/counts, raw JWT/session data, WorkOS/provider identifiers, backend component names, storage cursors, and correlation/idempotency internals beyond safe labels.

## `surface-audit-trace-detail` contract details

- Surface role: read-only evidence detail surface for one authorized Audit/Trace event, trace, denial, policy decision, provider/model/tool failure, support-access event, export decision, or workstream action. It explains what happened, why the actor is allowed to see the browser-safe subset, what was redacted, and which governed follow-up surfaces can be opened without mutating retained evidence.
- Owner: Audit/Trace workstream, functional agent `audit-trace-agent`. Reusable entry from dashboards, search rows, timelines, failure evidence, investigation guides, export requests, My Account, User Admin, Agent Admin, and Governance/Policy is by trace reference only; every open reauthorizes through Audit/Trace capabilities before any detail payload is returned.
- Actors and scope: tenant admins, tenant auditors/compliance reviewers, authorized support operators, and customer-scoped support actors with selected `AuthContext`, tenant id, optional customer scope, role/capability summary, and active support-access basis where required. Unauthorized, disabled-user, expired-support-access, cross-tenant/customer, hidden, purged, or retention-restricted evidence returns `not_found_or_redacted` or `forbidden` without confirming that the trace exists.
- User goal: inspect a single event's safe evidence, authorization basis, redaction rationale, related trace links, recovery guidance, and allowed next investigation actions while preserving least privilege and evidence immutability.
- Primary decision/action: review the detail and choose a governed next step such as timeline, failure evidence, investigation guidance, redacted export request, investigation note append, search return, or dashboard return. Success means the actor receives `audit.trace.detail.v1` with browser-safe evidence for the selected scope and trace/correlation refs; failure means a typed denial/redaction state with no hidden-id enumeration.
- Data source: backend-owned Audit/Trace repository/projection over retained audit events, work traces, agent/tool/model/provider traces, authorization denials, policy decisions, support-access evidence, export decisions, and source-surface correlation metadata. The browser must not hydrate details from search-row caches, fixture evidence, raw provider payloads, local storage, or frontend-derived authority.
- Mutability boundary: the detail surface is a `detail-edit` rendering pattern used as read-only evidence. It never edits source traces, source workstream records, policy decisions, retained audit rows, provider payloads, or authorization state. Investigation notes are separate annotations through `surface-audit-trace-investigation-note` and must be idempotent, scoped, and trace-linked.

### Detail payload schema

Top-level browser-safe fields for `audit.trace.detail.v1`:

- `surfaceContract`, `surfaceId`, `generatedAt`, `selectedScope`, `authContextSummary`, `capabilityIds`, `correlationId`, `traceRefs`, `redaction`, `readiness`, `detailKey`, `evidenceSummary`, `authorizationBasis`, `eventDetails`, `decision`, `source`, `relatedEvents`, `policyRefs`, `omittedFields`, `availableActions`, `emptyState`, `validationErrors`, and `recovery`.
- `detailKey`: safe display handle for the selected retained evidence, diagnostic trace label when authorized, event category, retention status, and stale/purged/redacted status. Raw storage keys, database ids, provider ids, hidden tenant/customer ids, and bearer/session identifiers are internal-only.
- `evidenceSummary`: timestamp/age, event kind, severity, status/decision, safe actor label/type, source workstream/surface label, correlation label, short redacted narrative, evidence category badges, and user-actionable next-step text.
- `authorizationBasis`: selected `AuthContext` summary, role/capability labels, support-access basis and expiry when applicable, customer-scope restriction, visible capability ids, and redaction explanation. Hidden policy implementation ids and missing-capability lists are not shown in normal browser copy.
- `eventDetails`: field groups for request/action metadata, authorization outcome, policy decision summary, provider/model/tool/runtime category, source surface/action label, and retained outcome. Values are redacted or omitted by category; raw prompts, raw tool payloads, credentials, invitation tokens, full email/message bodies, unredacted export bodies, and cross-scope evidence are never browser-visible.
- `decision`: safe result classification such as `allowed`, `denied`, `approval_required`, `blocked_provider_or_runtime`, `validation_error`, `recorded`, `not_found_or_redacted`, `partial_data`, or `failure`, plus safe reason and recovery text.
- `relatedEvents[]`: safe trace labels, correlation labels, event category, source workstream/surface, redaction badge, relationship type, and authorized action ids for timeline/detail/failure drilldowns. Hidden related events are summarized only as omitted categories when counts are safe.
- `policyRefs[]`: role-gated policy names or stable public policy references, decision summaries, and review links when allowed. Internal clause ids, policy engine internals, and hidden implementation names remain internal-only unless an auditor/support diagnostic role is explicitly authorized.
- `omittedFields` and `redaction`: omitted field keys by safe category, redaction reason, hidden-count policy, tenant/customer/support-scope restrictions, retention/purge explanation, and trace refs proving redaction was applied.
- `availableActions[]`: stable action id, browser label, capability id, target/result surface, approval requirement, idempotency/correlation expectation, and hidden/denied reason category for audit logs. Hidden actions are normally omitted; safe denied outcomes are typed system states rather than disabled authority leaks.

Default visible content is limited to investigation summaries, authorization basis labels, redaction explanations, trace/correlation labels, safe field groups, related-event summaries, and next-step actions. Role-gated auditor/support drilldowns may expose diagnostic trace ids, public policy refs, support-access basis, and source-system labels. Internal-only metadata includes raw event bodies, raw prompts, provider/model/tool payloads, credentials, WorkOS/provider identifiers, raw JWT/session data, full storage cursors, hidden tenant/customer identities/counts, backend component names, and correlation/idempotency internals beyond safe labels.

## Frontend-safe payloads

- Detail/failure evidence: `traceId`, `eventKind`, `timestamp`, `actor`, `source`, `authorizationBasis`, `decision`, `safeReason`, `redactedEvidence`, `relatedEvents[]`, `policyRefs[]`, `userActionableNextSteps[]`, and `redactionMetadata`.

- Investigation guide: `guideKey`, `investigationContext`, `riskSummary`, `recommendedPath[]`, `allowedActions[]`, `disabledActions[]`, `evidenceSummary`, `policyRefs[]`, trace links, redaction summary, and no authority-expansion marker.
- Export request: `exportId`, `status=approval_required`, `requestedFormat`, `reasonSummary`, `policyDecision`, `bundleMetadata`, allowed/disabled actions, trace links, and redaction summary.
- Investigation note: note summary, target trace id, idempotency key, retained-authority statement, and redaction metadata.
