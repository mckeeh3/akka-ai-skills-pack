# Global Roles: Foundation roles

Role definitions are aligned to the reusable foundation doctrine and current implementation role/capability model.

- `authenticated-member`: signed-in human with at least one active membership and selected `AuthContext`.
- `saas-owner-admin`: app-owner/platform operator who may administer other SaaS Owner Admin users, customer-facing Organizations backed by internal Tenant boundaries, and Organization Admin bootstrap/maintenance for those Organizations when the selected `AuthContext` is SaaS Owner scoped and backend capabilities such as `saas_owner.admin.*`, `saas_owner.organization.*`, `saas_owner.organization_admin.*`, platform-level `agent_admin.*`, or internal `saas_owner.tenant.read/manage` mappings are present. This role does not gain tenant/customer application-data access, support access, or billing-derived authority by implication.
- `tenant-admin`: Organization Admin for a customer-facing Organization/Tenant; manages tenant employees, customer records under that Organization, Customer Admin bootstrap/maintenance, invitations, role assignments, support access, and Agent Admin/managed-agent governance within authorized tenant scope. This role cannot manage sibling Organizations, SaaS Owner Admins, platform-level agents, or grant SaaS Owner authority.
- `tenant-employee`: non-admin Organization/Tenant member with canonical runtime role id `TENANT_EMPLOYEE`. Default authority is My Account, selected-context visibility, personal notifications/digest, and app-specific tenant workstreams explicitly granted by future business-domain capabilities. It cannot administer users, customers, roles, support access, Agent Admin, Governance/Policy, or Audit/Trace by default.
- `customer-admin`: manages customer users within the selected tenant/customer boundary. This role cannot create sibling customers, manage Organization Admins, administer Agent Admin/managed-agent behavior, or grant tenant/app-owner authority.
- `customer-user`: non-admin Customer-scoped member with canonical runtime role id `CUSTOMER_USER`. Default authority is My Account, selected customer context visibility, personal notifications/digest, and app-specific customer workstreams explicitly granted by future business-domain capabilities. It cannot administer Customer Admins, tenant employees, roles, support access, Agent Admin, Governance/Policy, or Audit/Trace by default.
- `auditor`: reads authorized audit and trace evidence without mutation authority; audit access does not grant Agent Admin access.
- `policy-owner-approver`: reviews, approves, activates, or rolls back governance and behavior changes within authorized scope only when backed by a SaaS Owner/App Admin platform-governance context or a tenant/organization-admin selected context for Agent Admin behavior.
- `agent-steward`: manages governed managed-agent behavior records and proposals only through a SaaS Owner/App Admin platform-governance context or tenant/organization-admin selected context with explicit `agent_admin.*` capabilities.
- `support-access-operator`: time-bound/support-scoped actor whose access is explicit, expiring, approved, and audited.

Backend roles/capabilities and selected context are authoritative; frontend visibility is never an authorization grant.
